  MEMBRE 1 — Diagnostic Adaptatif (Feature F1)  
Ta mission
Construire le questionnaire intelligent qui diagnostique la vraie maturité de l'entrepreneur — pas celle qu'il déclare, la réelle.
Ce que tu dois livrer
Un programme Python qui pose des questions et classe l'entrepreneur dans l'un des 6 stades
L'arbre de décision : les questions changent selon les réponses précédentes (branchement dynamique)
3 cas documentés de divergence perception-réalité (l'entrepreneur se croit au stade X, le système dit Y)
Le fichier JSON de sortie (contrat F1) — format fourni par le Membre 5

L'arbre de décision — logique de branchement
Exemple concret à implémenter :
Q1 : Dans quel secteur travaillez-vous ?
  → Agri-food : poser des questions sur certifications sanitaires, chaîne de froid, saisonnalité
  → Tech/Digital : poser des questions sur MVP, MRR, propriété intellectuelle
  → Industrie : poser des questions sur équipements, normes ISO, FOPRODI

Q2 : Avez-vous des revenus ?
  → OUI : Quel CA annuel ? Depuis combien de temps ?
  → NON : Avez-vous des clients payants ?
         → OUI : Lettres d'intention documentées ?
         → NON : Qui a validé votre idée ? → ALERTE divergence probable

  MEMBRE 2 — Scoring Multi-Dimensionnel (Feature F2)  
Ta mission
Construire le moteur qui calcule 5 scores sur le projet de l'entrepreneur et explique pourquoi — comme un vrai comité de crédit, pas une moyenne simple.
Ce que tu dois livrer
Un programme Python qui reçoit un profil JSON (F1) et retourne 5 scores détaillés
Un document de méthodologie : les poids, les critères, la logique — tout justifié
10 profils synthétiques avec leurs scores attendus (test set pour prouver la cohérence)
Le fichier JSON de sortie (contrat F2) — format fourni par le Membre 5

Les 5 scores obligatoires
Score

20%

20%

Réplicabilité / Dépendance manuelle / Coûts déploiement / Potentiel géo

Impact env / Viabilité écologique / Empreinte carbone / ODD / Éco-circulaire

Règle critique — Les scores ne sont PAS des moyennes
Un score faible sur une dimension fondamentale ne doit PAS être masqué par des scores forts ailleurs.

Exemple : Market Score = 10/100 → le système doit signaler clairement que ce projet
n'est PAS finançable même si les 4 autres scores sont à 80/100.

Logique à implémenter : si Market Score < 30 OU Commercial Score < 25
→ Financing Readiness Index plafonné à 40/100 peu importe le reste.

L'API centrale à construire (J3-J5)
# api_main.py — connecte tout

def run_pipeline(entrepreneur_input):
    # Étape 1 : Diagnostic
    diagnostic = f1_diagnostic(entrepreneur_input)

    # Étape 2 : Scoring (reçoit le diagnostic)
    scores = f2_scoring(diagnostic)

    # Étape 3 : RAG (reçoit diagnostic + scores)
    roadmap = f3_rag(diagnostic, scores)

    # Étape 4 : Tout envoyer au dashboard
    return { ...diagnostic, ...scores, ...roadmap }
